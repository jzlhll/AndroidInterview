## 一、Compose 架构（architecture）

Compose 架构以**声明式 UI**为核心，搭配**单向数据流（UDF）** 设计，彻底告别命令式 UI 的手动视图更新，UI 渲染完全由状态驱动，遵循 **UI = f(State)** 核心公式，状态变化自动触发界面重组，保证视图与数据一致性。

架构按**业务职责**分层，遵循关注点分离原则，共四层：

1. **UI 层**：专注界面展示与交互捕获，接收 ViewModel 提供的状态渲染 UI，将用户操作（点击、输入）转为事件向下传递，不处理任何业务逻辑。
2. **ViewModel 层**：作为 UI 与数据的中间层，持有并管理 UI 状态，处理业务逻辑，调用领域 / 数据层能力，生命周期独立于 UI，配置变更（如旋转屏幕）不丢失数据。
3. **领域层（可选）**：封装跨页面复用的复杂业务规则，解耦 UI 层与数据层，简化 ViewModel 逻辑，适合中大型项目。
4. **数据层**：负责数据获取（网络、本地数据库）、数据缓存、数据源管理，向上提供统一、稳定的数据接口。

数据流规则：**事件向下传递，状态向上回流**，单向流动避免数据冲突与状态混乱；核心遵循**状态提升**原则，将共享状态移至共同父组件，保证唯一可信源，提升可测试性与维护性。

## 二、Compose 分层（layering）

Compose 分层是**UI 框架的技术栈分层**，与业务架构分层是不同维度，核心目标是高内聚、低耦合，支持灵活定制与高效复用，上层基于下层 API 构建，可按需替换任意层。

### 分层 自底向上，底层为核心引擎

```
┌─────────────────────────┐
│ Material 3 层           │ 官方标准设计系统，提供成品组件、主题配置、标准化交互，快速搭建合规界面
├─────────────────────────┤
│ Foundation 层           │ 基础 UI 能力集，提供布局、Modifier、绘制、手势、动画基础 API
├─────────────────────────┤
│ UI 元素层               │ 原子 UI 组件，提供最基础的文本、图像、图形绘制能力
├─────────────────────────┤
│ Runtime 层              │ Compose 核心引擎，负责组合、重组、状态跟踪、布局渲染
└─────────────────────────┘
```

各层职责：

1. **Runtime 层**：框架运行基石，管理组合实例、触发精准重组、跟踪状态依赖、完成布局测量与绘制，保障 UI 高效更新。
2. **UI 元素层**：最基础原子组件，无样式封装，仅提供原始渲染能力，是构建自定义 UI 的最小单元。
3. **Foundation 层**：封装基础通用能力，Modifier 修饰符链、Row/Column/Box 布局、基础手势与动画均在此层，支撑高度自定义 UI 开发。
4. **Material 3 层**：顶层成品组件库，遵循 Material Design 规范，提供开箱即用的组件与主题，降低常规界面开发成本。

分层优势：底层稳定可靠，上层灵活扩展，职责边界清晰，既支持快速开发，也满足深度定制需求。
