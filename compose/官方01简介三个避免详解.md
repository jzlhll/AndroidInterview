Jetpack Compose 作为声明式 UI 框架，其核心优势在于高效重组，而实现这一优势的关键的是避免「副作用」，

**3个避免**：

* 避免写入共享对象的属性；
* 避免更新`ViewModel`中的可观察对象；
* 避免更新`SharedPreferences`等耗时操作且导致状态不一致问题

### 第一章 共享对象数据

共享对象通常指全局单例、顶层对象、静态变量或跨 Composable/页面共享的普通对象，这类对象的属性若直接在 Composable 函数体中修改，会导致重组混乱、性能浪费甚至状态错乱。

将共享对象的**属性通过 remember + mutableStateOf 托管**，使状态纳入 Compose 重组生命周期，避免与重组流程冲突。

共享对象属性的修改操作，脱离 Composable 重组流程，确保状态可控，必须放在**用户交互回调（如 onClick、onTextChange）或 Compose 副作用 API** 中，绝对不能直接写在 Composable 函数体中。

#### 一、错误示例（禁止直接修改共享对象属性）

以下代码直接在 Composable 函数体中修改全局共享单例的属性，违背「避免写入共享对象属性」的规范，会引发严重问题。

```kotlin
// 全局共享单例（跨页面/Composable可访问）
object GlobalConfig {
    var theme: String = "light" // 共享属性：应用主题
    var fontSize: Int = 16     // 共享属性：字体大小
}

@Composable
fun ThemeSettingScreen() {
    // ❌ 错误：在Composable函数体中直接修改共享对象属性
    GlobalConfig.theme = "dark"
    GlobalConfig.fontSize = 18

    Column(modifier = Modifier.padding(16.dp)) {
        Text(text = "当前主题：${GlobalConfig.theme}")
        Text(text = "当前字体大小：${GlobalConfig.fontSize}sp")
    }
}

```

问题分析：Composable 函数会被高频、不可预测地重组，每次重组都会重复执行上述赋值操作，造成无意义的性能浪费；同时，若其他页面也修改该共享对象，会出现读写冲突，导致 UI 渲染与实际状态不一致，甚至引发重组异常。

#### 二、正确示例（remember + mutableStateOf 状态托管）

```kotlin
// 全局共享单例（保持不变）
object GlobalConfig {
    var theme: String = "light"
    var fontSize: Int = 16
}

@Composable
fun ThemeSettingScreen() {
    // ✅ 核心：用remember缓存状态，避免重组重复创建；用mutableStateOf让状态可观察
    val currentTheme = remember { mutableStateOf(GlobalConfig.theme) }
    val currentFontSize = remember { mutableStateOf(GlobalConfig.fontSize) }

    Column(modifier = Modifier.padding(16.dp)) {
        Text(text = "当前主题：${currentTheme.value}")
        Text(text = "当前字体大小：${currentFontSize.value}sp")

        // ✅ 仅在用户交互回调中修改共享对象和托管状态
        Button(onClick = {
            // 修改共享对象属性
            GlobalConfig.theme = "dark"
            GlobalConfig.fontSize = 18
            // 更新Compose托管状态，触发UI重组
            currentTheme.value = "dark"
            currentFontSize.value = 18
        }) {
            Text("切换深色主题+放大字体")
        }
    }
}
```



### 第二章 ViewModel、LiveData 与 Flow

ViewModel 是 Compose 架构中状态管理的核心，负责持有页面级状态、封装业务逻辑；LiveData 和 Flow（尤其是 StateFlow）是常用的响应式数据流，用于实现状态的异步传递与监听。

官方规范要求：LiveData 和 Flow 需通过带有生命周期感知的扩展函数（xxxWithLifecycle）与 Compose 联动，确保状态监听安全、无内存泄漏，同时无需手动处理 remember 与生命周期绑定。

#### 一、ViewModel 基础正确用法

ViewModel 需封装私有可变状态，对外暴露只读状态，修改逻辑通过 ViewModel 内部方法实现，Composable 仅负责读取状态、触发事件。

Flow（尤其是 StateFlow）是 Kotlin 协程生态的响应式数据流，与 Compose 联动时，需使用 collectAsStateWithLifecycle 扩展函数，该函数内部已封装 remember，自动绑定页面生命周期（页面后台时暂停收集，销毁时取消订阅），避免内存泄漏，同时将 Flow 转换为 Compose 可观察状态，驱动 UI 重组。

LiveData 是 Android 传统响应式组件：与 Compose 联动时，官方推荐使用 observeAsStateWithLifecycle 扩展函数（替代旧版 observeAsState），该函数同样自带 remember 与生命周期感知，自动将 LiveData 转换为 Compose 可观察状态，确保监听安全。



```kotlin
// 1. ViewModel 封装状态与业务逻辑
class UserViewModel : ViewModel() {
    // 私有可变状态：仅在ViewModel内部修改
    private val _userName = MutableStateFlow("默认用户")
    private val _userAge = MutableLiveData(18)
    
    // 对外暴露只读状态：Composable仅能读取
    val userName: StateFlow<String> = _userName
    val userAge: LiveData<Int> = _userAge

    // 状态修改方法：Composable通过调用该方法修改状态
    fun updateUserName(newName: String) {
        _userName.value = newName
    }

    fun incrementAge() {
        _userAge.postValue(_userAge.value ?: 18 + 1)
    }
}

// 2. Composable 中使用 ViewModel
@Composable
fun UserProfileScreen(
    // 入参ViewModel，默认通过viewModel()获取
    userVM: UserViewModel = viewModel()
) {
    // 定义变量接收监听结果
    val userName by userVM.userName.collectAsStateWithLifecycle()
    val userAge by userVM.userAge.observeAsStateWithLifecycle(initialValue = 18)
    
    // 仅读取ViewModel暴露的状态，不直接修改
    Column(modifier = Modifier.padding(16.dp)) {
        Text(text = "用户名：$userName")
        Text(text = "年龄：$userAge")

        Button(onClick = { userVM.updateUserName("张三") }) {
            Text("修改用户名")
        }
        Button(onClick = { userVM.incrementAge() }) {
            Text("年龄+1")
        }
    }
}

```

### 第三章 SharedPreferences（SP）：正确使用方式

SharedPreferences 是 Android 轻量级键值对存储工具，用于存储应用配置、用户偏好等简单数据。由于其操作属于磁盘 IO 耗时操作，官方规范要求：避免在 Composable 函数体中直接执行 SP 的读写操作，需在用户交互回调或 Compose 副作用 API 中执行，同时通过 remember 缓存 SP 实例，避免重组重复创建。

#### 正确示例（用户交互回调中执行 SP 操作）

```kotlin
// SP工具类（单例，统一管理读写）
object SpUtil {
    private lateinit var sp: SharedPreferences

    // 初始化SP（在Application中调用）
    fun init(context: Context) {
        sp = context.getSharedPreferences("app_settings", Context.MODE_PRIVATE)
    }

    // 写入Boolean类型数据
    fun putBoolean(key: String, value: Boolean) {
        sp.edit().putBoolean(key, value).apply()
    }

    // 读取Boolean类型数据
    fun getBoolean(key: String, defaultValue: Boolean): Boolean {
        return sp.getBoolean(key, defaultValue)
    }
}

// Composable 中使用工具类
@Composable
fun SettingsScreen() {
    // 用mutableStateOf托管SP状态
    val isDarkMode = remember {
        mutableStateOf(SpUtil.getBoolean("dark_mode", false))
    }

    Column(modifier = Modifier.padding(16.dp)) {
        Text(text = "深色模式：${if (isDarkMode.value) "开启" else "关闭"}")
        Button(onClick = {
            val newMode = !isDarkMode.value
            // ✅ 调用工具类执行SP操作
            SpUtil.putBoolean("dark_mode", newMode)
            isDarkMode.value = newMode
        }) {
            Text(if (isDarkMode.value) "关闭" else "开启")
        }
    }
}

```

###  LaunchedEffect 与 onClick

前文多次提到，副作用操作需放在用户交互回调（如 onClick）或 LaunchedEffect 中执行，二者均能脱离 Composable 函数体、避免触发不必要的重组，但适用场景、执行时机和核心作用存在本质区别。

差异集中在「触发时机」「执行场景」和「生命周期绑定」上，具体如下：

- onClick 等回调：**用户主动触发**（如点击、输入），同步执行，无生命周期自动绑定，仅在用户操作时触发一次，适用于响应用户交互的副作用（如修改状态、SP 写入）。
- LaunchedEffect：**Composable 重组且满足条件时自动触发**，在协程中异步执行，与当前 Composable 生命周期完全绑定（组件销毁时自动取消协程），适用于无需用户触发的异步副作用（如网络请求、定时器、延迟操作）。

#### 二、用法示例

##### 1. onClick 回调（用户交互触发，同步执行）

适用于用户主动操作后执行的副作用，如修改状态、SP 读写、调用 ViewModel 方法，同步执行且无协程依赖。

```kotlin
@Composable
fun ClickCallbackDemo(vm: CountViewModel = viewModel()) {
    val count by vm.countFlow.collectAsStateWithLifecycle(initialValue = 0)

    Column(modifier = Modifier.padding(16.dp)) {
        Text(text = "计数：$count")
        // ✅ onClick：用户点击时触发，同步执行，无协程
        Button(onClick = {
            // 同步修改状态，无异步操作
            vm.increment()
        }) {
            Text("点击+1")
        }
    }
}
```

##### 2. LaunchedEffect（自动触发，异步执行，生命周期绑定）

适用于异步操作，需传入「key 参数」（控制触发时机），协程会在 Composable 进入组合时启动，组件销毁时自动取消，避免内存泄漏。

```kotlin
@Composable
fun LaunchedEffectDemo(vm: MessageViewModel = viewModel()) {
    val message by vm.message.observeAsStateWithLifecycle(initialValue = "加载中...")
    // 模拟：用户输入的搜索关键词（UI层状态，非后端返回）
    val searchKey = remember { mutableStateOf("") }

    // ✅ LaunchedEffect：Composable 进入组合时自动触发（初始化请求）
    // key = Unit：仅在组件首次进入组合时触发一次，调用ViewModel异步请求
    LaunchedEffect(key1 = Unit) {
        // 仅触发ViewModel的异步业务方法，不做任何业务判断（业务逻辑全部在VM）
        vm.fetchHomeMessage() // ViewModel内部处理后端请求、数据判断
    }

    // ✅ key = searchKey.value：当UI层搜索关键词变化时，重新触发协程（实际开发高频场景）
    // 核心：Compose层仅负责「触发异步操作」，业务逻辑（搜索请求、结果处理）全部在ViewModel
    LaunchedEffect(key1 = searchKey.value) {
        // 防抖处理（UI层专属逻辑，不属于业务，适合放在LaunchedEffect）
        delay(500) // 输入防抖，避免频繁请求
        // 调用ViewModel搜索方法，将关键词传入，业务判断由VM负责
        vm.searchMessage(searchKey.value)
    }

    Column(modifier = Modifier.padding(16.dp)) {
        // 搜索输入框，修改searchKey，触发上面的LaunchedEffect
        TextField(
            value = searchKey.value,
            onValueChange = { searchKey.value = it },
            label = { Text("输入关键词搜索消息") }
        )
        Spacer(modifier = Modifier.height(16.dp))
        Text(text = message, fontSize = 18.sp)
        // 手动触发刷新，仅调用VM方法，无业务判断
        Button(onClick = { vm.fetchHomeMessage() }) {
            Text("刷新首页消息")
        }
    }
}

// 对应ViewModel（业务逻辑全部封装，Compose不参与任何判断）
class MessageViewModel : ViewModel() {
    private val _message = MutableLiveData("加载中...")
    val message: LiveData<String> = _message

    // 异步请求首页消息（业务逻辑在VM）
    fun fetchHomeMessage() = viewModelScope.launch {
        // 模拟后端请求
        delay(1000)
        val response = "首页消息加载完成（后端返回）"
        // 业务判断、数据处理全部在VM内部
        _message.value = if (response.isNotEmpty()) response else "暂无消息"
    }

    // 搜索消息（业务逻辑在VM）
    fun searchMessage(keyword: String) = viewModelScope.launch {
        if (keyword.isEmpty()) {
            _message.value = "请输入搜索关键词"
            return@launch
        }
        // 模拟后端搜索请求
        delay(800)
        val searchResult = "搜索到与「$keyword」相关的3条消息"
        _message.value = searchResult
    }

    // 仅提供状态修改方法，供Compose调用
    fun updateMessage(newMsg: String) {
        _message.value = newMsg
    }
}

```

#### 三、关键注意事项

- LaunchedEffect 必须传入 key 参数，key 不变则协程仅执行一次；key 变化时，会取消旧协程、启动新协程。
- 无需用户触发的异步操作（如页面初始化请求、定时器），必须用 LaunchedEffect；用户交互相关的同步操作，用 onClick 等回调即可。
- 二者均能避免副作用污染 Composable 函数体，但 LaunchedEffect 额外解决了异步操作的生命周期管理问题，避免协程泄漏。



### 第五章 总结

Jetpack Compose 官方「3个避免」的核心，是确保 Composable 函数仅做「纯渲染」（读取状态、输出UI），杜绝函数体中出现副作用，保障重组高效、状态一致。

所有状态修改、耗时操作（如SP读写），需放在用户交互回调或 LaunchedEffect 中；共享对象用 remember+mutableStateOf 托管，ViewModel 封装状态与修改逻辑，LiveData/Flow 用 xxxWithLifecycle 联动，遵循单向数据流即可规避副作用。
